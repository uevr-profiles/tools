# Discord API Logic

## Metadata Concept
The Discord metadata is generated by scraping a Discord Forum channel using the Discord REST API for messages.

**Endpoint Forms**: `https://discord.com/api/v10/channels/{channel_id}/messages` or `https://discord.com/api/v10/threads/{thread_id}/messages`
**Headers Required**: 
- `Authorization: Bot {TOKEN}` (or a user token if self-botting/authorized apps)
- `Content-Type: application/json`

The Discord API acts as the data structure source, returning an array of Message objects. An intermediary bot script iterates over forum threads looking for messages that possess `.zip` file attachments.

**Example Message Attachment Snippet**:
```json
{
  "attachments": [
    {
      "id": "1214040927690199040",
      "filename": "UEVR_Profile.zip",
      "size": 14210,
      "url": "https://cdn.discordapp.com/attachments/1111/2222/UEVR_Profile.zip",
      "content_type": "application/zip",
      "description": "My Profile setup"
    }
  ]
}
```

## Download Concept
**Endpoint Form**: `https://cdn.discordapp.com/attachments/{channel_id}/{message_id}/{filename}` (This is exactly the string given in the `url` property of the attachment array)
**Headers Required**: Typical browser `User-Agent`

**Implementation Constraints**:
1. Discord Attachment CDN URLs are fully absolute. They are provided intrinsically by the `url` property of the `attachments` array in the Metadata pass.
2. No explicit API authentication or token header is passed (or required) to download from the Discord CDN once you possess the direct link to the attachment URL.
3. Depending on recent Discord changes to attachment security or deleted messages/channels, the CDN may return a `404 Not Found` or `403 Forbidden` response. Download routines must anticipate these HTTP errors gracefully without treating them as catastrophic connection failures.
